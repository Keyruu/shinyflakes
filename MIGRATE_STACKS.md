# Migrating Existing Stacks to the Stack Module

This document describes how to migrate existing quadlet stack files to use the
new `services.my.<name>.stack` module and `flake.lib.quadlet` helpers.

## Reference: Migrated Example

See `nix/hosts/mentat/modules/stacks/karakeep.nix` for a fully migrated stack.

## Step-by-Step Migration

### 1. Update the let block

Remove `stackPath` and `quadletToService`. Replace with `quadlet`.

```nix
# Before:
let
  stackPath = "/etc/stacks/service-name";
  my = config.services.my.service-name;
  inherit (config.virtualisation.quadlet) containers;
  inherit (flake.lib) quadlet;
in

# After:
let
  my = config.services.my.service-name;
  inherit (config.virtualisation.quadlet) containers;
  inherit (flake.lib) quadlet;
in
```

- Remove `stackPath` — use `my.stack.path` instead everywhere
- Remove `networks` from inherit if the stack uses `stack.network.enable`

### 2. Delete manual boilerplate

Remove these sections entirely — they are auto-generated by the stack module:

- **`systemd.tmpfiles.rules`** — replaced by `stack.directories`
- **`virtualisation.quadlet.networks.<name>.networkConfig`** — replaced by
  `stack.network.enable = true`
- **`backup.paths = [ stackPath ]`** — auto-wired from `stack.path`
- **`backup.systemd.unit = ...`** — auto-wired from `stack.containers.members`
- **`serviceConfig.Restart = "always"`** on every container — already the
  quadlet-nix default, remove all occurrences
- **`users.users` / `users.groups`** blocks — replaced by `stack.user`
- **`networks = [ networks.<name>.ref ]`** on each container — auto-wired when
  `stack.network.enable = true`

### 3. Add stack configuration to `services.my`

Replace the old `backup = { ... }` block with the simplified form and add the
`stack` block:

```nix
services.my.service-name = {
  # keep existing: port, domain, proxy
  backup.enable = true;  # simplified — paths and units are auto-wired
  stack = {
    enable = true;
    # each string becomes a tmpfiles rule: "d /etc/stacks/<name>/<dir> ..."
    directories = [ "data" "config" ];
    # only for multi-container stacks:
    network.enable = true;
    # pass container attrsets directly (NOT .ref strings)
    containers = with containers; {
      members = map quadlet.service [
        service-main
        service-db
      ];
      security.enable = true;
    };
  };
};
```

If the stack had custom users/groups, add:

```nix
stack.user = {
  enable = true;
  name = "myuser";     # defaults to service name
  uid = 1000;          # or null for auto-assigned
  # group defaults to user.name
  # gid = 1000;        # or null for auto-assigned
};
```

### 4. Update references throughout the file

#### Volume paths

Replace `${stackPath}/...` with `${my.stack.path}/...`:

```nix
# Before:
volumes = [ "${stackPath}/data:/data" ];
# After:
volumes = [ "${my.stack.path}/data:/data" ];
```

#### SOPS template restartUnits

Replace `quadletToService` with `quadlet.service`:

```nix
# Before:
restartUnits = map quadlet.service [ containers.service-main containers.service-db ];
# After:
restartUnits = with containers; map quadlet.service [ service-main service-db ];
```

#### Inter-container hostnames

Replace hardcoded hostnames with `quadlet.alias`:

```nix
# Before:
MEILI_ADDR = "http://meilisearch:7700";
# After:
MEILI_ADDR = "http://${quadlet.alias containers.karakeep-meilisearch}:7700";
```

This only works when the target container has `networkAliases` set. The function
returns the first alias from `containerConfig.networkAliases`.

#### Remove network references from containers

Since the stack module auto-wires the network into all members:

```nix
# Before:
containerConfig = {
  ...
  networks = [ networks.karakeep.ref ];
  networkAliases = [ "meilisearch" ];
};

# After:
containerConfig = {
  ...
  # networks is auto-wired by stack.network
  networkAliases = [ "meilisearch" ];
};
```

### 5. Flatten the virtualisation.quadlet block

Since network config is handled by the stack module:

```nix
# Before:
virtualisation.quadlet =
  let
    inherit (config.virtualisation.quadlet) networks;
  in
  {
    networks.service-name.networkConfig = {
      driver = "bridge";
      podmanArgs = [ "--interface-name=service-name" ];
    };
    containers = { ... };
  };

# After:
virtualisation.quadlet.containers = { ... };
```

### 6. Remove redundant serviceConfig

`Restart = "always"` is already the quadlet-nix default. Remove it from every
container:

```nix
# Before:
karakeep-web = {
  containerConfig = { ... };
  serviceConfig = {
    Restart = "always";
  };
  unitConfig = { ... };
};

# After:
karakeep-web = {
  containerConfig = { ... };
  unitConfig = { ... };
};
```

## Special Cases

### Stacks with host networking (e.g. home-assistant)

Do NOT set `stack.network.enable` or `stack.containers.members` for containers
using `networks = [ "host" ]`. The host network is not a quadlet network and
cannot be auto-wired. Keep the `networks = [ "host" ]` on the container
directly.

```nix
stack = {
  enable = true;
  directories = [ "config" ];
  # no network, no members
};
```

### Stacks with custom directory permissions

If the old tmpfiles.rules had custom ownership or modes:

```nix
# Before:
systemd.tmpfiles.rules = [
  "d ${stackPath}/data 0755 1000 1000"
  "d ${stackPath}/mariadb/config 0755 root root"
];

# After:
stack.directories = [
  { path = "data"; mode = "0755"; owner = "1000"; group = "1000"; }
  { path = "mariadb/config"; }  # defaults to 0755 root root
];
```

When `stack.user` is enabled, string entries default to user ownership with
`0750`. To keep root ownership for specific dirs, use the attrset form.

### Stacks with security exceptions

If a container needs capabilities (e.g. `NET_ADMIN`), add them on the container
directly — the stack module uses `mkDefault` so explicit values win:

```nix
containers = with containers; {
  members = [ home-assistant ];
  security = {
    enable = true;
    # or exclude it entirely:
    exclude = [ home-assistant ];
  };
};

# On the container itself:
containerConfig = {
  addCapabilities = [ "NET_ADMIN" "NET_RAW" ];
};
```

### Stacks where the function argument list changes

If `networks` is no longer needed in the function args, and `flake` was only
used for `quadletToService`, check if `flake` is still needed (for `quadlet`).
If SOPS templates are present, `flake` is still needed for `quadlet.toService`.
If there are no SOPS templates and no `quadlet.*` usage, `flake` can be removed
from the function args.

## Checklist

For each stack file:

- [ ] Remove `stackPath` let binding, use `my.stack.path`
- [ ] Replace `inherit (flake.lib) quadletToService` with
      `inherit (flake.lib) quadlet`
- [ ] Remove `inherit (config.virtualisation.quadlet) networks` if using
      `stack.network`
- [ ] Delete `systemd.tmpfiles.rules`
- [ ] Delete `users.users` / `users.groups` blocks
- [ ] Delete manual `networks.<name>.networkConfig`
- [ ] Delete `backup.paths` and `backup.systemd.unit`
- [ ] Delete all `serviceConfig.Restart = "always"`
- [ ] Delete `networks = [ networks.<name>.ref ]` from each container
- [ ] Add `stack = { enable = true; ... }` block
- [ ] Replace `${stackPath}` with `${my.stack.path}` in volumes
- [ ] Replace `quadletToService` with `quadlet.service`
- [ ] Replace hardcoded inter-container hostnames with `quadlet.alias`
- [ ] Flatten `virtualisation.quadlet = let ... in { ... }` to
      `virtualisation.quadlet.containers = { ... }`
- [ ] Verify the file has no LSP errors
